# TDR-030: Hybrid instance licensing — local unlimited, cloud metered per seat

## Context

ainative's distribution model assumes power users will run multiple domain-focused instances on the same machine (wealth manager, investor relations, growth/sales, etc.), each with its own `STAGENT_DATA_DIR` and branch. The license row lives in the SQLite settings table per data dir, so architecturally every instance has its own tier activation.

This creates a commercial design question: *how should licensing work when a single paying user runs 3 clones on their laptop?*

Three options were explored during the clone lifecycle brainstorm:

1. **Strict: 1 paid seat = 1 instance.** Each clone needs its own activation key. Creating instance #2 prompts "buy another seat." More revenue, but creates friction exactly where power users want freedom — the activation UX for instance #2 is indistinguishable from "this is a different user" so there's no natural path to offer upgrade pricing. Also breaks the current architecture because `LicenseManager.activate()` doesn't know about "instances," only about one local license row per data dir.

2. **Generous: 1 seat = 1 machine, unlimited instances.** A single paid license activates all local data dirs automatically. Simplest UX but no upsell path: a team sharing a laptop under one seat could run as many instances as they want, indistinguishable from a single user. The current architecture almost supports this already — two clones with the same email and `STAGENT_CLOUD_DISABLED=true` both work as scale tier.

3. **Hybrid: local features unlimited, cloud features metered per seat.** Local tier features (tasks, workflows, documents, agents, the task runner, the entire offline value prop) work in any number of `STAGENT_DATA_DIR`s without gating. Cloud-side features (cloud sync, marketplace, telemetry, license validation itself) meter seats at the Supabase edge function using a stable `(email, machineFingerprint, instanceId)` tuple.

Option 3 matches the product principle that ainative's offline/local value is the primary pitch and cloud features are a complement. It also matches the existing architecture: `LicenseManager` already reads the license row per data dir, so no local gating work is needed. Only the cloud validation payload needs to change.

## Decision

Instance licensing is **hybrid**:

- **Local features are always unlimited per machine.** The `LicenseManager` continues to read the `license` row from its own data dir and apply tier-based feature gates locally. No seat count check. No cross-instance coordination. A user with 3 clones on scale tier gets the full scale feature set on all 3.

- **Cloud features meter seats via `(email, machineFingerprint, instanceId)`.** `LicenseManager.validateAndRefresh()` sends this tuple to the Supabase edge function during daily cloud validation. The edge function enforces `seats(email).localInstances <= tier.maxInstances` (community: 1, solo: 2, operator: 5, scale: unlimited). On seat-over-limit, the cloud returns `seatStatus: "over_limit"` which is stored in `settings.license.seatStatus` — the local tier is NOT downgraded, but the over-limit banner surfaces in the Settings → License UI with an upgrade CTA.

- **`machineFingerprint`** is a stable, non-identifying hash computed from `os.hostname() + os.userInfo().username + sha256(primary network MAC)`. It survives clone removal + reinstall on the same machine and provides a stable coordinate for seat counting without revealing any personal data.

- **`instanceId`** is a per-clone UUID generated by `ensureInstanceConfig()` on first boot and stored in `settings.instance`. It survives `Clear All Data` (the settings table is preserved) and distinguishes sibling clones on the same machine.

- **Seat counting is advisory client-side, authoritative server-side.** The client has `TIER_LIMITS.maxInstances` only for UI hints ("You have 3 of 5 instances — upgrade for unlimited"). The actual enforcement happens at the edge function. This means the server can be built, updated, or temporarily degraded without client changes.

## Consequences

- **Zero local gating work** for instance count. `LicenseManager.isFeatureAllowed()` continues to check tier → feature flags only; it never counts instances.
- **Existing paying users are not affected** because `STAGENT_CLOUD_DISABLED=true` short-circuits cloud validation entirely, so the tuple is never sent and the seat check is never performed. The three private domain clones on this machine (wealth, investor, growth) continue to work on scale tier.
- **The Supabase edge function becomes the authoritative seat counter** and can evolve independently of clients. Over-limit handling (7-day grace, soft banner, upgrade CTA) is a server-side policy decision.
- **Machine migration is smooth**: when a user gets a new laptop and re-clones, the `machineFingerprint` changes but the `instanceId` stays the same if they restore their data dir. The cloud can detect the mismatch and treat it as a seat migration, not a new instance (policy TBD in the edge function).
- **Revocation is possible** (future): the cloud can mark a specific `instanceId` as revoked, and the next validation round will surface the state in `settings.license.seatStatus`. Local features continue to work (local doesn't enforce), but cloud features degrade and the UI can prompt for re-activation.
- **Clean separation of concerns**: local license code doesn't know or care about instance counts; instance code doesn't know or care about tiers.
- **The upsell path is clear**: community users hit `maxInstances = 1` early and see a clear upgrade prompt when they create instance #2. Solo users hit it at instance #3. Operator at #6. Scale users never hit it. The tier ladder maps naturally to instance counts for power users without forcing normal users into an explicit seat purchase flow.

## Alternatives Considered

- **Strict per-instance activation** (option 1 above) — rejected. High friction, high implementation cost, no clear upsell pricing. Would require a dedicated "activate another instance" flow with its own UI, which duplicates the existing activation flow without meaningful differentiation.
- **Generous unlimited** (option 2 above) — rejected. No upsell path. Also inconsistent with the tier philosophy where premium tiers should offer more capability, not just the same capability with different limits elsewhere.
- **Honor system, zero enforcement** — rejected for the same reason as generous unlimited. Works for individual hobbyists but doesn't support a revenue model.
- **Counting instances entirely client-side via some coordination mechanism** (shared lock file in `~/.ainative-meta/`, process discovery, etc.) — rejected. Fragile, hackable, and requires cross-instance IPC which is a whole new surface area.
- **Per-email cloud seat count ignoring machineFingerprint** — rejected because it would conflate "same user on two machines" (legitimate and should probably be free for scale tier) with "two instances on one machine" (what we're trying to meter).
