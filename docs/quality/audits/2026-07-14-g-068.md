# G-068 mutation-strength and pruning disposition

Date: 2026-07-14
Scope: database integrity, workflow execution, schedule claims, runtime target
truth, Chat finalization, Pack provenance, licensing, and bounded duplicate
candidate review.

## Decision

G-068 establishes a reusable dependency-free mutation gate and performs **no
test pruning**. No remaining candidate met all five pruning-policy requirements
(same invariant/provenance, equal boundary fidelity, no relevant coverage loss,
equal-or-better fault detection, and measured maintenance/flake/runtime
improvement). The measured pruning reduction is therefore zero; forcing a
deletion would contradict the goal.

The goal instead adds two guards discovered by mutation evidence:

- Operations Receipt `source_key` index uniqueness, separate from the older
  index-presence assertion
- persisted workflow-child failure propagation, separate from the broader
  top-level workflow failure fixture

## Reproducible control

`npm run test:mutation-strength` copies current `src` and Vitest configuration
into a marked repo-local disposable shell, shares the existing dependency tree
for module resolution while redirecting Vitest's cache, and gives every child
fixture-owned HOME/USERPROFILE, APPDATA, LOCALAPPDATA, XDG, cache, and temporary
roots. Its allowlisted child environment contains no credential variables or
live profile locations. The harness writes mutations only to the disposable
source copy. This is process-environment isolation, not an operating-system
filesystem sandbox. Every source anchor must match exactly once before the
baseline begins.

A non-zero test run counts as a kill only when the normalized verbose output
contains a `FAIL` line for the mutant's named assertion. Timeout, signal,
collection/setup failure, unrelated test failure, missing/ambiguous anchor,
restore mismatch, and cleanup failure are named harness errors.

### Weak-assertion discovery

The first valid matrix before the workflow assertion was added killed 6/7
required mutants. `workflow-failed-child-propagates` survived while the broad
workflow suite stayed green. G-068 therefore added the direct persisted-child
result assertion and narrowed the manifest to that authoritative test. The
repaired matrix kills the fault. This was treated as evidence of a missing guard,
not as a reason to change the mutation or inflate the score.

The intentional DB control mutates `CREATE UNIQUE INDEX` to `CREATE INDEX` and
runs only the historical presence/idempotency assertion. It survives as
designed. The paired full-strength assertion inspects SQLite's `unique` index
metadata and kills the identical mutation. This proves the runner reports weak
assertion selection instead of equating green coverage with strength.

## Repeatability and runtime

Four consecutive repaired runs produced the same 7/7 kills, one expected
survivor, 63/63 baseline/restoration counts, and cleanup result. Total durations
were 8,631 ms, 8,277 ms, 8,363 ms, and 8,257 ms (median 8,320 ms; range 374 ms).
No flake or outcome drift occurred. The bounded command remains below the
goal's existing local verification budget and adds no dependency.

## Rollback

The package command, manifest, runner, two regression assertions, audit fields,
and documentation are additive. Reverting the G-068 commit removes the control.
No production behavior or schema changed, and every deliberate source mutation
occurred only inside a verified disposable copy.
