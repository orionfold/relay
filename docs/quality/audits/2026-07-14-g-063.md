# G-063 Relay regression audit

**Audit date:** 2026-07-14
**Code baseline:** `d1665c992445` plus preserved unrelated worktree changes
**Verdict:** Relay has strong regression islands, but its load-bearing areas are
not completely protected. The 3,000+ tests do not justify broad pruning.

## Questions answered

### Are all load-bearing areas 100% regression covered?

No. “100%” is meaningful only as complete mapping of agreed critical
invariants, not repository-wide line coverage. Packs, licensing, instance
bootstrap, cryptography, and several data contracts are strong. Workflow
execution, real runtime adapters/module loading, Chat engines/SSE routes, API
mutations, scheduler concurrency, test-database safety, release gates, and
critical browser/runtime journeys have material gaps.

### Can the 3,000+ tests be pruned?

Only narrowly on present evidence. Two task-schema assertions in the original
setup-verification file duplicate the authoritative adjacent validator suite and
form the bounded pilot. Provider contract tests and source-policy tests can
probably be consolidated, but their scenarios must remain until mutation,
runtime, and timing evidence proves a stronger replacement. No evidence
supports deleting hundreds or thousands of tests.

### Is the harness/strategy current?

Partly. Vitest 4, V8 coverage, Testing Library, real SQLite integration, and a
separate E2E concept remain sound. The configuration had not caught up with
Vitest 4 coverage and pool changes; all Vitest tests defaulted to jsdom; the
default suite collected server-dependent E2E; three `node:test` suites lived
outside that inventory; no risk thresholds or general CI gate exist;
browser/runtime checks are not hermetic or required; and environment isolation
can target an inherited data directory.

## Methodology

- Static inventory through `rg`, filesystem enumeration, test/config/CI review,
  and git history.
- Full Vitest JSON run with wall-clock timing.
- Explicit production-surface V8 coverage run, preserving reports on failure.
- One fixed-seed shuffled run plus isolated reruns of newly failing files.
- Per-file/per-test timing from Vitest JSON.
- Independent two-pass static review using the project code-review checklist.
- Disposable inherited-`RELAY_DATA_DIR` probe.
- Official Vitest, Testing Library, Playwright, and GitHub Actions guidance.
- Bounded before/after pruning pilot with deliberate fault injection across the
  validator plus representative database, schedule, runtime, Pack, licensing,
  and Chat-stream guards.

Counts from `scripts/test-audit.mjs` are source inventory. Executed test counts
come from Vitest JSON; parameterized and conditional tests make the numbers
intentionally different.

## Isolation and flake evidence

`src/test/setup.ts` creates a temporary data directory only when
`RELAY_DATA_DIR` is absent. A safe probe set `RELAY_DATA_DIR` to
`/tmp/g063-inherited-data` and ran the slot-claim suite. The test wrote an 872 KB
`relay.db` and a 3.9 MB WAL directly into that inherited directory. This proves
the harness cannot currently promise it will never mutate a developer/customer
database.

The fixed-seed shuffle exposed six additional failures. The E2E configuration
also used the removed Vitest 3 `poolOptions.forks.singleFork` option. G-063
migrates that option to Vitest 4's `maxWorkers: 1, isolate: false` equivalent but
does not broaden the E2E contract.

Static E2E inventory finds five test files. Invoking the E2E config after the
migration emits no deprecation, but `vitest list` exposes only the unconditional
blueprint case because provider-dependent cases are conditionally registered.
The default config's `vitest list` also exited zero while reporting an
`EnvironmentTeardownError` from an entity-detector import after environment
teardown. Both are harness diagnostics defects: enumeration must be reliable,
and a required lane must not disappear before execution reporting.

Slowest baseline files included instance bootstrap (7.999 s), desktop prebuilt
download (5.182 s), plugin transport dispatch (3.106 s), CLI env handling
(2.370 s), upgrade polling (2.226 s), and Pack install/update suites (~1.5–2.1
s). The slowest single test was the desktop HTTP download case at 5.004 s.

## CI and release topology

- `.github/workflows/fresh-clone-dev.yml` is path-filtered and runs hooks plus
  four selected Vitest files and one boot smoke.
- `.github/workflows/publish.yml` runs on version tags and tests only licensing
  and desktop before building/smoking/publishing.
- No workflow runs the default full suite, production-surface coverage, fixed-seed
  shuffle, typecheck, or critical runtime/browser lane on general pull requests.

This is the highest-leverage modernization gap: a strong local test that is not
required before merge/release is advisory, not a gate.

## Pruning ledger

| Candidate | Classification | Evidence / action |
|---|---|---|
| Two task-schema assertions in setup verification | remove (pilot) | duplicated by the adjacent task-validator suite; setup file retains runner and alias checks |
| Task validator suite | keep | owns boundary/default/deprecated-field/status/error behavior |
| DB bootstrap/migration tests | keep | real SQLite and recovery behavior; load-bearing history |
| License verify/store tests | keep | strong cryptographic/tamper/term/entitlement protection |
| Pack install/update/provenance tests | keep | high-value real round trips; no mass consolidation evidence |
| OpenAI/Anthropic direct MCP merge tests | consolidate later | same provider contract shape can use a shared parameterized contract; retain provider-specific serialization/errors |
| Runtime adapter tests that mock Chat tools | strengthen/move | keep unit payload scenarios, add real-module-graph smoke before considering deletion |
| Scheduler “concurrent” claim test | relabel/strengthen | currently proves idempotent sequential refusal; add two-connection/barrier test |
| Source-text architecture/policy tests | consolidate later | replace multiple regex assertions with one typed architecture checker, then retain the checker |
| Broad component suite | keep pending risk map | no mutation/timing evidence supports deletion; migrate interactions selectively |
| Snapshot tests | no action | none exist |

The safe quantified reduction is deliberately small: two duplicate executed
cases (about 0.06% of the baseline) and their maintenance surface. Separating 12
server-dependent E2E cases removes them from `npm test` but does not delete them.
Provider-contract consolidation may reduce source duplication, not scenario
count; no runtime reduction is claimed until measured.

## Bounded pruning pilot

Before the change, setup verification plus the authoritative task-validator
suite executed 26/26 tests and covered `src/lib/validators/task.ts` at 100% lines,
functions, statements, and branches. The pilot removes only the setup file's
valid/invalid task examples; runner execution and `@/` alias resolution remain.
After the change, the same slice executed 24/24 tests with the same 100% line,
function, statement, and branch coverage. Raising the accepted priority maximum
from 3 to 4 made the retained `rejects priority above 3` assertion fail; exact
restoration returned the slice to green. The removed cases therefore added no
unique validator path or fault detection.

## Modernization decisions

### Retain

- Vitest 4 and V8 coverage.
- React Testing Library for deterministic component behavior.
- Real SQLite/filesystem integration tests where state integrity matters.
- Separate runtime, browser, and customer-identical staging layers.
- Zero snapshot files; add snapshots only for a deliberate stable contract.
- Strong Pack, licensing, bootstrap, and cryptographic suites.

### Change in G-063

- Explicitly include `src` and the shipped CLI in coverage, retain only named
  shell/test exclusions, and report on failures.
- Exclude server-dependent E2E from default collection while retaining its
  named command.
- Migrate the removed Vitest single-fork option.
- Add a dependency-free, JSON-capable test inventory command.
- Adopt the risk policy in `docs/quality/regression-strategy.md`.

### Defer to bounded goals

- Harness-owned data directories and shuffled-run determinism.
- General PR/release quality gates and risk-tiered thresholds.
- Real runtime/module-graph/Chat-stream smoke and concurrent SQLite claims.
- Vitest projects for Node versus jsdom, selective `user-event`, and pinned real
  browser assertions.
- Mutation sampling and any further consolidation/pruning.

Official guidance supports these directions: Vitest 4 recommends an explicit
coverage include and replaced `singleFork`; Vitest projects support different
environments; fixed-seed shuffle finds hidden order dependencies; Testing
Library prioritizes user-observable semantic interaction; Playwright recommends
isolated tests and web-first assertions; and GitHub's Node workflow guidance
places build/test commands in CI.

### Post-audit resolution — G-022

G-022 closed the red baseline without changing production behavior or adding an
allow-failure list. The five deterministic failure families were stale or
incorrect test boundaries: the privacy audit followed strategy-repository
symlinks, router tests mocked the pre-dispatch runtime layer, the desktop test
required a non-hermetic loopback listener, Web Designer still expected two tables
after Web Publisher added `web_pages`, and auth validation still expected the
method field after model-only partial updates shipped. The two shuffle failures
were test-order leaks: row history retained a parent-table foreign key, and a
throwing scanner mock implementation survived `vi.clearAllMocks()`.

After targeted repair, the default suite and shuffled seeds `6301`, `1729`, and
`90210` each executed 3,225 cases with 3,224 passed, zero failed, and one
intentional live-app conditional skip. The targeted deterministic slice passed
51/51 and the isolated randomized-order slice passed 13/13.

### Post-audit resolution — G-064

G-064 removed the demonstrated destructive test boundary. Before the change, a
passing schedule database test run with a disposable inherited
`RELAY_DATA_DIR` silently created `relay.db`, `relay.db-shm`, and `relay.db-wal`
beside its sentinel. The default Vitest configuration now creates a marked
temporary root during global setup, replaces inherited application state before
worker imports, assigns a distinct child directory to each worker, and restores
stubbed environment/global state after every test. Global teardown restores the
inherited environment; a synchronous parent-process exit hook removes the root
after worker-held database handles close, and only after its temporary-directory
location, name, regular-directory type, marker schema, and creation nonce all
pass ownership checks.

The dedicated `npm run test:harness-safety` child-process guard passed while a
worker exercised real SQLite: an inherited `relay.db` sentinel remained
byte-for-byte unchanged and the reported harness root did not exist after
process exit. The entire default suite repeated that proof with the inherited
sentinel and executed 3,232 cases: 3,231 passed,
zero failed, and one intentional live-app conditional skip. Fixed-seed shuffle
`6301` produced the same result. Seven harness regressions cover pre-import
override, marker/schema checks, unsafe and forged paths, nonce mismatch,
per-worker uniqueness, Windows-safe path components, bounded cleanup, and
test-local environment/global restoration. TypeScript, diff, audit, doc-link,
and public-boundary checks were green at completion.

### Post-audit resolution — G-066

G-066 replaced mocked or mislabeled Tier-0 evidence with executable production
boundaries. The schedule claim matrix now releases two independent SQLite
connections through a shared barrier beneath cap 1 and proves exactly one
winner; the old same-connection double call is correctly labeled idempotency.
The source-wide TDR-032 guard found and removed two pre-existing static
`ainative-tools` imports in the Anthropic Direct and OpenAI Direct adapters,
while retaining dynamic call-site loading.

Ollama Chat now participates in the same termination contract as the primary
engine. Completed, upstream-error, nil-body, empty-body, signal-abort, client
cancel, iterator-abandonment, and stale-reconciliation paths all leave terminal
assistant rows and record their named reason codes. Delta-then-EOF and malformed
NDJSON are explicitly failed unless Ollama emits its `done: true` terminal
frame. Client cancellation now
propagates into the generator's abort signal instead of leaving the provider
reader alive behind a closed SSE stream.

`npm run test:runtime-graph` passed against an isolated real Next/Turbopack
process and local deterministic Ollama transport. The child receives an
allowlisted environment plus fixture-owned Unix, Windows, and XDG profile roots,
so no parent credentials or configuration are available to a misrouted runtime.

`f635746c-adbe-45c9-b2b2-5faa655d449e` persisted requested/effective runtime
`ollama`, the exact task sentinel, and `started`/`completed` logs. Workflow
