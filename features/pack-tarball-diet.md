## Measurement — recorded 2026-07-06 (the entry criterion, done)

`npm pack --dry-run` on the current tree: **2.22 MB packed / 6.79 MB unpacked, 1129 files.**
The pack templates are **206 KB unpacked across 9 packs (~23 KB each)** — about **3%** of the
unpacked payload, negligible in the gzipped tarball. Projection at ~23 KB/pack: 25 packs ≈ 0.56 MB,
100 packs ≈ 2.24 MB unpacked. **Conclusion:** the tarball-bloat trigger Pillar F named is not
present at the current catalog size; cutting the bundled set now would add a network hop +
offline-fragility to the hot path to save ~150 KB — a bad trade today. The diet begins to pay for
its complexity around ~500 KB of templates (≈ 22 packs, roughly doubling today's footprint).

**So this session ships the MECHANISM, not the cut** (operator-decided 2026-07-06): the explicit
bundled-set SSOT + an automated size gate that FAILS the release when templates cross the budget —
turning "don't cut yet" from a forgotten deferral into an auditable, triggered decision.

## Built (mechanism half)

- **`src/lib/packs/bundled.ts`** — `BUNDLED_PACK_IDS` (the explicit "what ships bundled" allowlist,
  zero-import leaf) + `BUNDLED_TEMPLATES_SIZE_BUDGET_KB = 500` (the deferral trigger) +
  `isBundledPack()`. Mirrored to **`bundled.json`** (generated by
  `scripts/generate-bundled-json.mjs`, kept in lockstep by `bundled.test.ts`) so the plain-`node`
  gate can read the SSOT without importing `.ts` (the taxonomy pattern).
- **`scripts/check-pack-tarball.mjs`** (`npm run check:pack-tarball`) — fails CLOSED on two axes:
  (i) allowlist drift (a pack on disk not declared, or declared but missing — the `files`-allowlist
  trap), and (ii) size overflow (templates exceed the budget → force the cut). Wired into
  `npx-prod-smoke.mjs` as **Case TB** (runs at release, alongside Case T/P).

## Deferred (the cut + fetch-then-cache) — real remaining dependency surfaced

The spec's "cache a fetched long-tail pack into the managed `base/`" presumes the version-pinned
`base/` + git-SHA-lock overlay slot that `relay-core-and-packs.md §5` **explicitly deferred** as "a
later, additive spec" (v1 install is editable-seed). So the fetch-then-cache half is gated on that

# Pack tarball diet — slim default set + fetch-on-install (R4)

## Description

`packs-robustify.md`'s explicit target is **hundreds of packs**. Today every pack ships as a
bundled template in the npm payload, so every customer downloads every pack, forever — the
tarball-bloat trigger Pillar F named. This feature resolves it with the split the remote resolver
(R2) makes possible: a **slim bundled default set** (the most-installed packs, resolving offline
with zero network, exactly as today) + **fetch-on-install for the long tail** (rare/vertical
packs fetched via the R1 index on first `pack add`, then cached into the managed `base/`).

The design is deliberately *not* "fetch everything" (that breaks offline-first and adds a network
hop to the hot path) and *not* "bundle everything" (the bloat). It is a measured cut: bundle the
packs a first-run customer reaches for immediately (the free `relay-agency` spine + the top
premium packs); fetch the rest on demand. Subsequent boots read the cache; only `pack update`
re-fetches. Fail-OPEN offline for a bundled pack (never needs the network); fail-LOUD for a
long-tail pack the customer explicitly asked to install and that cannot be reached (Principle #1).

This is **P2 and gated on prerequisites**: it needs R2's fetch mechanism shipped *and* a real npm
tarball-size measurement + a most-installed cut (**open decision #2**), coordinated with the
Website peer. This spec is the *implementation* of the split once the cut is decided — the
measurement + the boundary decision are its entry criteria, called out explicitly so grooming
never treats R4 as buildable before R2 lands and the cut is measured.

## User Story

As a first-run Relay customer, I want the npm download to stay small (only the packs I'm likely
to use immediately ship bundled), while any other pack installs on demand from the canonical
index and then works offline forever after — so the catalog can grow to hundreds of packs without
my download growing with it.

## Technical Approach

### Entry criteria (must be true before building)

1. **R2 shipped** — `resolvePackSourceAsync` + the sha-verified fetch exist (this spec reuses
   them wholesale for the long-tail path).
2. **A real tarball-size measurement** — measure `npm pack` size with the current bundled set;
   quantify the per-pack cost so the cut is evidence-based, not guessed.
3. **The slim/fetch boundary decided (open decision #2)** — which pack ids stay bundled vs. fetch.
   Needs a most-installed signal + Website-peer coordination (`strategy/relay/_RELAY.md`).

### The split mechanism

- **Bundled set** — unchanged: ships in `src/lib/packs/templates/`, resolved local-first by the
  sync `resolvePackSource`, zero network. A curated allowlist (a small data file, e.g.
  `BUNDLED_PACK_IDS` or read from what's physically in `templates/`) is the SSOT of "what ships."
- **Long-tail set** — physically removed from the npm tarball (deleted from `templates/` or
  excluded via the `files` allowlist in `package.json`), present only in the canonical index. On
  first `pack add <long-tail-id>`, R2's remote branch fetches it, sha-verifies it, and caches it
  into the managed **base/** (the overlay model's version-pinned base,
  `relay-core-and-packs.md §5` — the lock file records the git sha; `pack update` re-pulls).

### Caching into the managed base/ (the overlay graduation)

A fetched long-tail pack must **cache**, not re-fetch every boot. The overlay model already has
the slot: the version-pinned `base/` with a sha lock. R4 wires the fetch to *write into* that
base on first install (the "graduation of base-vendoring from copy-from-local to
fetch-from-GitHub" the doc names), so subsequent installs/boots read the cache. `pack update` is
the only path that re-fetches (bumps the lock sha).

### The `files` allowlist trap (regression guard)

Recall `0.32.1`: `public/brand/` had to be added to the `files` allowlist or npx 404'd the asset
(memory `logo-3d-swap-recipe`). The inverse risk here: if a long-tail pack is deleted from
`templates/` but the `files` glob still references it, or if a *bundled* pack is accidentally
excluded, the npx prod smoke breaks. Update the npx prod-smoke Case L pack-install counts to
match the new bundled set (memory `prod-smoke-encodes-contracts` — Case L asserts install counts;
a bundled-set change is a smoke-count site).

### Verification

Runtime-registry-adjacent (touches install + `package.json` `files`) → **dev-server smoke**:
`npm pack` size drops measurably; a bundled pack installs offline; a long-tail pack fetches +
caches on first `pack add` and installs offline on the second; the npx prod smoke Case L counts
pass with the new set.

      exceed the budget — the deferral trigger that will surface "now perform the cut."

**The cut + fetch-then-cache (deferred — gated on the size trigger + the §5 overlay graduation):**
- [ ] A long-tail pack (removed from the tarball, present only in the index) fetches +
      sha-verifies + caches into the managed `base/` on first `pack add`, then installs offline on
      subsequent runs. *(Blocked on the `relay-core-and-packs.md §5` managed-base overlay slot.)*
- [ ] A long-tail pack that cannot be reached fails **loudly**; a bundled pack never triggers a

      offline behavior.)*

- [ ] **Dev-server smoke** proves the full bundled-offline + long-tail-fetch-then-cache loop.

## Scope Boundaries

**Included:**
- The measured cut + the bundled/long-tail split implementation.
- Caching a fetched long-tail pack into the managed `base/` (overlay graduation).
- Updating the npx prod-smoke Case L counts + the `files` allowlist.

**Excluded (separate requirements / gated):**
- **The fetch mechanism itself** → `pack-remote-resolver.md` (R2). R4 *uses* it; it does not
  rebuild it.
- **Deciding the slim/fetch boundary** → open decision #2 (a measurement + a most-installed cut +
  Website coordination). This is an *entry criterion*, not built here.
- **Provenance** of a fetched long-tail pack → `pack-provenance-tiers.md` (R3) already verifies
  any fetched pack.
- Any change to the overlay `base/overrides` model itself (`relay-core-and-packs.md §5`) — R4
  writes into the existing slot, it does not redesign it.
