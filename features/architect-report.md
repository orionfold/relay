# Architect Report

## Integration analysis — G-108 portable Relay Host Linux VM playbook

### Accepted boundary

G-108 adds a release-versioned deployment surface beside Relay Core:

```text
orionfold-relay npm release
  → portable manifest + checked public bootstrap assets
  → relay-host-playbook render/preflight/verify CLI
  → customer-created compatible Ubuntu VM
  → protected Host activation using existing ingress/licensing/recovery paths
```

It does not add a provider API client, credential store, infrastructure state,
Fleet Controller, online entitlement service, provider branch in Relay Core, or
new application UI. TDR-044 remains authoritative without amendment.

### Integration surfaces

| Surface | G-108 disposition |
|---|---|
| npm Host distribution | adds exact deploy assets, customer guide and CLI to the package allowlist |
| Cell authority | consumes the existing signed digest-pinned Cell release manifest unchanged |
| Host lifecycle | reuses the accepted supervisor, ingress, entitlement and recovery contracts |
| provider ownership | customer creates, pays for and removes provider resources; Relay retains no provider credential |
| Website | receives customer-friendly portable-playbook copy; public adaptation/publication remains Website/operator-owned |
| Core/data/API/UI | unchanged |

### Security findings and resolutions

- Cloud user-data accepts only a public SSH key and lowercase hostname label.
  Secret-bearing fields, secret-like values and unknown inputs fail closed.
- The manifest rejects unknown fields and pins Relay/Cell authority, bootstrap
  bytes, Node and Cosign versions/checksums.
- Bootstrap separates the passwordless-sudo `relayadmin` SSH operator from the
  locked non-sudo `relay` runtime identity. No default second login user is
  created by the template.
- Bootstrap prepares public artifacts but does not expose Relay, install a
  license or receive ingress/model/recovery secrets.
- Both deliberate refusals and unexpected shell failures write mode-0600 named
  receipts. The local Ubuntu fixture proves a failed-disk receipt and identical
  retry success.
- Docker access remains part of the already accepted same-Host trust boundary;
  G-108 does not claim hostile isolation from the Host administrator.

overlap. No unresolved critical or informational finding remains. Browser proof

Host regression and Ubuntu fixture evidence is the stronger guard.
