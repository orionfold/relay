# Architect Report

## Change Impact Analysis — canonical managed Relay Cell identity

### Proposed change

Resolve `RELAY_CELL_ID` through one application validator and give it precedence
over git-instance bootstrap in the read-only Cell-boundary contract.
Readiness keeps its `local` fallback only when no managed ID exists. Existing
`instanceId` response fields remain compatible.

### Blast radius

| Layer | Files affected | Impact |
|---|---|---|
| Configuration | `src/lib/config/env.ts` | one named validator/resolver for the managed Cell environment contract |
| Boundary domain | `src/lib/instance/cell-boundary.ts` | managed identity precedes dev/no-git/git fallback decisions |
| Health API | `src/app/api/health/ready/route.ts` | consumes the shared validator and retains readiness response shape |
| Instance API/UI | existing config route and `InstanceSection` | no schema change; corrected value flows through current contract |
| Execution previews | existing task/workflow target routes and shared preview | no route change; corrected value flows through current context builder |
| Packaging | `scripts/relay-host-entry.mjs` plus artifact smoke | entrypoint grammar remains independent but must stay parity-tested |

**Classification:** medium — configuration, domain, API, UI, and packaging
verification are affected, but the implementation is localized to two runtime
resolvers plus their API/UI consumers and introduces no data migration.

### Dependency trace

`RELAY_CELL_ID` → shared environment resolver → `getRelayCellBoundary()` →
instance config + task/workflow execution contexts → Settings and execution
preview. The same resolver feeds readiness with an unmanaged `local` fallback.

### Migration requirements

- Database/schema/bootstrap migration: no.
- API versioning: no; retain `instanceId` and the readiness payload.
- Feature flag: no; this corrects an accepted managed-Cell invariant.
- TDR: no new decision. TDR-044 already makes identity part of each Cell and
  requires the image to accept a Cell manifest identity.

### Risks and mitigations

- Invalid environment values could silently fall through to git identity:
  throw a named `InvalidRelayCellIdError` and keep readiness's stable
  `CELL_ID_INVALID` reason.
- An import could make readiness load instance/bootstrap state: keep the shared
  resolver in the lightweight config layer; readiness must not import the full
  Cell-boundary module.
- Unmanaged npx/dev behavior could change accidentally: retain explicit absent,
  dev, no-git, git-backed and environment-precedence regressions.
- Entrypoint and application regexes could drift: assert the shared grammar in
  unit tests and rerun the real G-093 container contract.

### Recommended approach

Proceed with the compatible resolver. Do not rename `instanceId`, add a second
persisted identity, or expand into Host routing. After targeted tests and static
checks, rebuild the linux/arm64 artifact and verify the same Cell ID through
readiness, Settings and an execution-target preview.
