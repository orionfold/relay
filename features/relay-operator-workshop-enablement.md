# Relay Operator Workshop Enablement

## Outcome

Relay supplies the runnable product half of the first autonomous Orionfold
Training product: **Relay Operator Workshop — turn one operating memo into a
governed workflow**.

The learner can understand the result before buying, prove a machine is ready,
start from a deterministic Marketing Line fixture, adapt one real process into a
user-owned Relay app/workflow, add a human checkpoint and cost/trust boundary,
run automated checks, recover from known failures, and retain a portable
capstone plus evidence. The known-good path requires no founder teaching,
grading, scheduling, routine rescue, or fulfillment action.

This is a product-enablement program, not an LMS. The free Relay core remains
useful and sufficient for the capstone. Paid value belongs to the complete
guided workshop edition, diagnosis/retry paths, reference solution, completion
bundle, and Website fulfillment—not to disabling ordinary Relay capability.

## Product thesis

The highest-leverage workshop investment improves Relay itself and then reuses
that improvement in instruction and marketing:

1. A richer adaptive Home makes the product legible in screenshots, trailers,
   the static demo, and the learner's live workspace.
2. Semantic Render views make capstone data feel like an operating application
   instead of a spreadsheet.
3. The existing Marketing Line pack supplies the canonical demonstration.
4. The existing Chat app builder, workflows, human checkpoints, cost controls,
   operations receipts, output documents, and app-to-pack exporter supply the
   capstone mechanics.
5. The existing `_ASSETS` corpus supplies versioned product truth, docs,
   screenshots, synthetic fixtures, memos, and the static demo.

The new work should bridge those capabilities. It should not rebuild them under
workshop-specific names.

|---|---|---|

| Relay `_ASSETS` | current product truth, source/evidence fragment, Marketing Line memo, docs/API guidance, synthetic seed, screenshots, static sample lane, release freshness | paid access state, Motion masters/renditions, a second book pipeline |
| Website / Training owner | offer, syllabus, price/refund promise, guest checkout, access delivery, public sample, transactional messaging, later optional customer library | Relay runtime truth, audiovisual generation, duplicated product docs |
| Orionfold Motion | source snapshot, approved audiovisual job, long/short/vertical/instruction renditions, captions, provenance, technical QC, export bundle | curriculum truth, checkout, learner progress, capstone validation |
| Books | doctrine and durable PDF/EPUB/HTML editions | workshop runtime, duplicated product instructions |
| Orionfold Proof | receipt design precedent: evidence, verdict, reproducibility, honest limitation | a required network/service dependency or a second Relay receipt engine |
| Arena demo | static-simulator quality bar and deploy-rewrite precedent | copied product truth or a second Relay demo source |

The Training product owns the cross-product learning-unit envelope. Relay emits
one versioned, source-hashed Relay fragment from `_ASSETS` containing product
version, approved source references, starter artifact identity, capstone
criteria, demo-lane references, and known rescue paths. Motion and Website
consume a snapshot of that fragment; they do not scrape Relay internals.

## Experience contract

The Relay portion of the workshop follows this path:

1. **Inspect:** see the final governed workflow and its retained artifact.
2. **Preflight:** verify Relay version, local data directory, runtime
   availability, fixture integrity, and required free-core capabilities.
3. **Start:** install or clone the source-hashed starter into an isolated,
   clearly labeled local project/app.
4. **Adapt:** replace the supplied process with one bounded learner process.
5. **Govern:** add at least one human checkpoint, runtime/trust choice, and cost
   or budget signal.
6. **Run:** execute the workflow and watch truthful progress.
7. **Validate:** evaluate deterministic criteria first and an explicit bounded
   rubric only where judgment is unavoidable.
8. **Recover:** show the failed boundary, exact evidence, retry action, and
   known-good fallback without silently fixing or hiding the failure.
9. **Retain:** export the user-owned app/pack, output documents, and completion
   evidence with hashes and version identity.

The UI may expose a bounded workshop/checkpoint presentation, but it must not
grow a generic course catalog, social profile, remote progress service,
certificate system, points/streaks, cohort calendar, or instructor inbox.

## Source and freshness contract

The Relay workshop source fragment is derived from:

- `_ASSETS/memos/marketing-line/`;
- `_ASSETS/docs/` and `_ASSETS/api/`;
- `_ASSETS/seed/` and `_ASSETS/demo/`;
- the current bundled Marketing Line pack;
- current Relay package version and relevant feature/catalog hashes.

Generated scripts, Motion videos, Website pages, emails, and slides are
renditions. They never become source for the next edition.

Any product-version mismatch, missing source hash, unsupported workshop edition,
tampered starter, stale known-good fixture, or unavailable required runtime
fails visibly with a named error and a bounded rescue path. Relay must never
silently substitute current product behavior for a workshop edition that was
authored against another version.

## Locked W0 decisions

- **Delivery:** Website hosts the paid instruction and fulfills an account-free
  downloadable edition manifest/starter reference. Paid prose does not ship in
  Relay's public npm package.
- **Integrity:** every edition is content-addressed. External purchased
  editions may carry a detached Ed25519 signature verified against an explicit
  trusted-key registry. Built-in official sample/starter content is pinned by
  an embedded canonical hash.
- **Local state:** Relay persists one minimal workshop-run anchor with edition,
  checkpoint and evidence identities. It does not persist course catalogs,
  lessons, enrollment, cohorts, certificates or instructor state.
- **Completion language:** existing Operations Receipt verdicts remain
  authoritative: `passed`, `at_risk`, and `failed`. The workshop UI presents
  these as Completed, Completed at risk, and Needs work without changing their
  machine values.
- **Sharing:** local export is learner-owned and redacted. Any public receipt,
  badge, social share or Website rendering is opt-in and separately owned.

See TDR-045 for the architecture decision and
`features/relay-operator-workshop-enablement-plan.md` for implementation order,
regression budget and rescue/rollback.

## “WOW” and sharing policy

The workstream earns attention through visible capability and checkable proof:

- a dense but understandable Home command center;
- data rendered as a useful application, not raw rows;
- one memo becoming a runnable governed workflow;
- a real human checkpoint and cost/trust boundary;
- a before/after capstone preview;
- a portable artifact and rerunnable evidence bundle.

Do not add confetti, dark-pattern sharing prompts, fabricated activity, vanity
badges, forced signup, customer-data screenshots, or automatic social posting.
Permissioned learner sharing is a later Website/Marketing concern and must use
redacted/synthetic or explicitly approved evidence.
